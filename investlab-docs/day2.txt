SECTION 1 вЂ” Feature Overview
---------------------------------------------------------
What InvestLab is:
InvestLab is a real-time, risk-free paper trading simulator integrated directly into BudgetBuddy. It provides users with virtual cash to practice buying and selling actual stocks using live market data.

Why it exists inside BudgetBuddy:
BudgetBuddy helps users manage expenses and save money. InvestLab takes the logical next step: teaching users how to grow those savings. By evaluating a user's "Investment Readiness" based on their budgeting habits, it bridges the gap between saving and wealth building.

Student use case & Educational purpose:
Students and beginners often fear losing money in the stock market. InvestLab provides a sandbox environment to learn market dynamics, understand portfolio valuation, and practice emotional discipline without any financial risk.

Difference between paper trading and real trading:
Paper trading uses simulated money but real-time market data. Trades execute instantly at the current market price without broker fees, liquidity constraints, or actual financial loss. Real trading involves actual capital risk, slippage, and brokerage execution delays.

Why this feature adds value to BudgetBuddy:
It transforms a standard expense tracker into a comprehensive financial literacy platform. It encourages users to stick to their budgets (to improve their Readiness Score) and gamifies the learning process of investing.

SECTION 6 вЂ” Custom Hooks Deep Dive
---------------------------------------------------------
useMarketPrices.js:
- Purpose: Connects to Socket.io and maintains a dictionary of live prices.
- Socket Connection: Initializes `io('/', { path: '/socket.io' })`.
- Listeners: Listens for `priceSnapshot` (initial load) and `priceUpdate` (incremental ticks).
- Cleanup: Returns `() => socket.disconnect()` in the `useEffect` to prevent memory leaks when the user navigates away.

```javascript
// snippet
socket.on('priceUpdate', (update) => {
  setPrices((prev) => {
    const next = { ...prev };
    Object.keys(update.data).forEach(sym => {
      next[sym] = update.data[sym]?.price || next[sym];
    });
    return next;
  });
});
```

usePortfolio.js:
- Purpose: Fetches the user's portfolio data securely.
- API Flow: Uses Axios with `Authorization: Bearer ${user.token}`.
- State: Returns `{ portfolio, loading, error, refreshPortfolio }`. `refreshPortfolio` allows child components to trigger a refetch after a successful buy/sell.

```javascript
export const usePortfolio = () => {
  const { user } = useAuth();
  const [portfolio, setPortfolio] = useState(null);
  
  const fetchPortfolio = useCallback(async () => {
    const res = await axios.get('/api/investlab/portfolio', {
      headers: { Authorization: `Bearer ${user.token}` }
    });
    setPortfolio(res.data);
  }, [user]);

  useEffect(() => { fetchPortfolio(); }, [fetchPortfolio]);
  return { portfolio, refreshPortfolio: fetchPortfolio };
};
```

useReadiness.js & useTradeHistory.js:
Follow the exact same pattern as `usePortfolio`, leveraging Axios and `useAuth` to hit their respective endpoints.

SECTION 8 вЂ” API Integration
---------------------------------------------------------
GET /api/investlab/portfolio
- Purpose: Fetches cash balance and holdings.
- Detailed Flow:
  1. Frontend: `usePortfolio` hook calls `axios.get` with the user's JWT.
  2. Backend Middleware: `protect` middleware decodes the JWT and sets `req.user`.
  3. Backend Controller: `getPortfolio` queries MongoDB `Portfolio.findOne({ userId })`.
  4. Backend Logic: Maps through the holdings, calculates `holdingsValue` and `totalPnL` dynamically based on baseline prices.
  5. Response: JSON object containing cash and arrays of holdings.
  6. Frontend UI: Component updates, `PortfolioMetrics` renders total values.

POST /api/investlab/buy
- Purpose: Executes a buy order.
- Detailed Flow:
  1. Frontend: User fills `BuyStockCard` form. Component sends `{ symbol, quantity, price }` via POST.
  2. Backend Middleware: JWT is verified.
  3. Backend Controller: `buyStock` executes. Queries user's `Portfolio`.
  4. Backend Logic: Checks if `cashBalance >= requiredCost`. If sufficient, deducts cash. If holding exists, updates `quantity` and recalculates a weighted `avgBuyPrice`. If new, pushes to `holdings` array.
  5. Backend DB: Saves `Portfolio` document. Creates a new `Trade` document (type: 'buy').
  6. Response: HTTP 201 Created.
  7. Frontend Callback: Form receives 201, sets success state, calls `onBuySuccess()`.
  8. Frontend Refetch: `InvestLabPage` calls `refreshPortfolio()` and `refreshTrades()`. Data flows back down the React tree to instantly update `HoldingsTable` and `PortfolioMetrics`.

SECTION 10 вЂ” Real-Time Architecture
---------------------------------------------------------
WebSockets provide full-duplex, persistent communication over a single TCP connection, unlike REST which requires a new HTTP request for every interaction. 
Socket.io is a library that wraps WebSockets, providing automatic reconnections, fallbacks, and room-based broadcasting.
In InvestLab, REST is used for transactional data (buying/selling) because it is synchronous and guarantees success/failure states. WebSockets are used for market prices because they require high-frequency, low-latency continuous streaming.

SECTION 11 вЂ” Live Price Pipeline
---------------------------------------------------------
1. Finnhub (External): Pushes raw JSON trade data to our backend via WebSocket.
2. finnhubService.js: Parses the data, caches the latest price, and emits a Node.js `EventEmitter` event.
3. marketDataGateway.js: Listens to the emitter and publishes the data to Redis Pub/Sub.
4. server.js: Listens to Redis and broadcasts `io.emit('priceUpdate')` to all connected frontend clients.
5. useMarketPrices.js: Receives the Socket.io event and updates React state.
6. React UI: Re-renders the Watchlist and HoldingsTable instantly.

SECTION 12 вЂ” State Management
---------------------------------------------------------
Local State (`useState`): Used in components for UI-specific data (e.g., input values in forms).
Hook-based State: We created custom hooks (`usePortfolio`) to encapsulate state and share it via props from a top-level parent (`InvestLabPage`).
Why not Redux? Redux introduces significant boilerplate. Since InvestLab data is mostly server-state (needs to be fetched and mutated on the server), simple custom hooks + prop drilling (1 level deep) is much more efficient and maintainable.

SECTION 13 вЂ” Error Handling
---------------------------------------------------------
API Failures: Wrapped in `try/catch`. The `catch` block reads `err.response?.data?.message` (provided by Express) and sets it to local state, rendering an error banner in the UI.
Socket Failures: Socket.io automatically attempts to reconnect. The hook tracks a `connected` boolean.
Validation: Forms disable the submit button if inputs are empty or quantities are negative.

SECTION 16 вЂ” System Design Concepts
---------------------------------------------------------
Separation of Concerns: Hooks handle data logic; Components handle UI.
Event-Driven Architecture: Finnhub pushes events -> backend reacts -> frontend reacts. The system does not waste resources polling.
API Layering: The frontend doesn't talk to the DB; it talks to Express controllers, which talk to services, which talk to Mongoose.

SECTION 17 вЂ” Design Decisions
---------------------------------------------------------
Why Socket.io? It handles edge cases like dropped connections and CORS effortlessly compared to native WebSockets.
Why separate InvestLab components? Keeps the codebase modular. If InvestLab breaks, the core BudgetBuddy expense tracker remains unaffected.
Why keep routing unchanged? By putting it under `/stocks` and inside the `<MainLayout>`, the user experiences a seamless Single Page Application (SPA) feel without full page reloads.

SECTION 19 вЂ” Possible Interview Questions
---------------------------------------------------------
Q: How did you handle the high frequency of WebSocket updates in React?
A: By utilizing functional state updates (`setPrices(prev => {...})`) to prevent stale closures, and mapping the data carefully to avoid triggering massive DOM reflows unnecessarily.

Q: What happens if the Finnhub API goes down?
A: The backend implements exponential backoff for reconnections. To ensure the frontend doesn't break, I implemented a simulated fallback generator on the backend that emits small randomized price ticks when the market is closed or the connection drops.

Q: Why didn't you use Redux for the prices?
A: Stock prices change multiple times per second. Pumping that much volatile data through Redux reducers would cause massive overhead. Local hook state passed directly to the necessary components is much more performant.
