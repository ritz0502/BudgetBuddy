# INVESTLAB DAY 3 - TECHNICAL DOCUMENTATION

## SECTION 1 — DAY 3 OVERVIEW

### Intended Achievements
Day 3 was designed to transform InvestLab from a basic paper-trading simulator into an intelligent, AI-powered financial education platform. The focus shifted from mere transaction handling to providing actionable, personalized financial coaching, visualizing future wealth trajectories, and assessing portfolio health dynamically.

### Transformation into an Educational Platform
Instead of just letting users buy and sell stocks blindly, Day 3 introduced a layer of "Financial Intelligence." It leverages Google's Gemini AI to analyze user financial habits (savings rate, budget adherence) alongside their active portfolio to provide hyper-personalized advice. It also introduced a deterministic Rules Engine to deliver structured educational content and background processing to track historical performance.

### High-Level Features Implemented
1. **AI Financial Coach**: A Gemini-powered integration that ingests user financial metrics and portfolio data to generate personalized strengths, weaknesses, and actionable learning recommendations.
2. **Portfolio Allocation Analytics**: A Recharts-based visualization that dynamically calculates and renders the distribution of assets (stocks vs. cash).
3. **Portfolio Performance Tracking**: A historical time-series chart showing the growth or decline of the portfolio over time.
4. **Portfolio Snapshot System**: A background worker architecture (setInterval + BullMQ) that takes periodic snapshots of user portfolios for historical tracking.
5. **Future Wealth Planner**: A deterministic financial simulator using SIP (Systematic Investment Plan) compound interest formulas to project wealth trajectories over 10-50 years.
6. **Learning Center**: A Rules Engine-based educational module that conditionally displays specific financial lessons (e.g., Budgeting, Diversification) based on the user's Readiness Score and portfolio concentration.
7. **Portfolio Health Score**: A comprehensive algorithmic score assessing diversification, cash reserves, concentration risk, and readiness alignment.
8. **UI Restructure**: A 6-section dashboard overhaul prioritizing high-level analytics over raw tables.

## SECTION 3 — GEMINI AI INTEGRATION

### Architecture

```text
User 
  ↓ (Clicks Refresh)
AI Coach Card (Frontend)
  ↓ (GET /api/investlab/ai-coach)
InvestLab Controller
  ↓ (Compiles Context)
Gemini Service (generateCoachInsights)
  ↓ (Network Request)
Gemini API (Google)
  ↓ (Returns Markdown JSON)
Response Processing (JSON.parse + Clean)
  ↓ (JSON Object)
Frontend Rendering
```

### How Gemini is Called
- **File**: `backend/services/geminiService.js`
- **Prompt**: Uses a heavily structured prompt instructing the model to act as a financial advisor and return *only* a 5-field JSON object.
- **Data Sent**: Readiness Score, Savings Rate, Budget Adherence, Portfolio Value, Holdings Summary, Concentration Risk, and Recent Trades.
- **Data Received**: A JSON string wrapped in markdown blockticks (\`\`\`json ... \`\`\`).
- **Parsing**: The service strips the markdown backticks using regex `.replace(/```json\s*/, '').replace(/```\s*$/, '')` and calls `JSON.parse`.

### Error Handling & Fallback
If Gemini fails (e.g., 429 Quota Exceeded), the `catch` block in the controller triggers `generateFallbackInsights()`. This is a deterministic function that mimics the AI's output format perfectly, ensuring the frontend never crashes.

### Important Code Snippets

**Prompt Construction & API Call (`geminiService.js`)**
```javascript
const prompt = `You are an expert financial advisor...
Analyze the following data:
Readiness Score: ${context.readinessScore}
...
Return EXACTLY AND ONLY a valid JSON object...`;

const result = await client.models.generateContent({
  model: 'gemini-2.5-flash',
  contents: prompt,
});
const raw = result.text;
// Parsing
const cleaned = raw.replace(/```json\s*/, '').replace(/```\s*$/, '').trim();
return JSON.parse(cleaned);
```
*Explanation*: 
1. We inject user variables using template literals.
2. We call `generateContent` targeting the fast `gemini-2.5-flash` model.
3. We extract the text, aggressively strip markdown formatting (which LLMs notoriously inject even when told not to), and parse it into a native JavaScript object.

**Rules Engine**. It is NOT AI-based. It is deterministic, personalized, and conditional.

```text

     ↓
```

### Why Rules Engine Was Chosen
- **Predictability**: We always want the exact same lesson taught for the exact same mistake. AI hallucinates.
- **Cost**: 0 API calls. Free.
- **Performance**: 0ms latency. Calculated instantly on the frontend.
- **Reliability**: Cannot fail due to rate limits or network errors.

## SECTION 5 — FUTURE WEALTH PLANNER

### Purpose
To visualize the power of compounding interest over decades based on the user's specific monthly investment capabilities.

### User Flow
1. User enters **Monthly Investment**, **Investment Style (Annual Return %)**, and **Years**.
2. Frontend sends `POST /api/investlab/projection`.
3. Backend loops through every year, applying the SIP Future Value formula.
4. Returns an array of yearly data points and milestones.
5. Frontend renders an AreaChart.

### Mathematical Formula
**SIP Compound Interest Formula**:
`FV = P * [((1 + r)^n - 1) / r] * (1 + r)`
Where:
- `P` = Monthly Investment
- `r` = Monthly Return Rate (Annual Rate / 12)
- `n` = Total number of months

### Worked Example
- ₹3000/month
- 10% Annual Return
- 10 years (120 months)
- Monthly Rate `r` = 0.10 / 12 = 0.008333
- `FV = 3000 * [((1 + 0.008333)^120 - 1) / 0.008333] * (1 + 0.008333)`
- `FV ≈ ₹6,19,656`

### Code Snippets (`investlabController.js` - getProjection)
```javascript
const monthlyRate = r / 12;
for (let year = 0; year <= n; year++) {
  const months = year * 12;
  // ...
  const futureValue = P * ((Math.pow(1 + monthlyRate, months) - 1) / monthlyRate) * (1 + monthlyRate);
  projectionData.push({ year, value: futureValue });
}
```
*Explanation*: 
We loop through each year, calculate the total months passed, apply the Future Value of a Series formula, and push the data point into an array used directly by Recharts.

## SECTION 6 — PORTFOLIO HEALTH SCORE

### Purpose
A single numerical metric (0-100) evaluating the structural safety and risk profile of a portfolio.

### Scoring Factors
1. **Diversification (40%)**: 1 holding = 20 pts, 4+ holdings = 100 pts.
2. **Cash Reserve (30%)**: 10-30% cash = 100 pts. Too little or too much drops the score.
3. **Readiness Alignment (30%)**: Matches the user's BudgetBuddy Readiness Score.
4. **Concentration Risk (Penalty)**: If a single stock is >50% of the portfolio, subtract 15 points from the final score.

### Formula & Example
`Overall = (Div * 0.4) + (Cash * 0.3) + (Align * 0.3) - Penalty`
Example:
- Diversification: 3 holdings (70) -> 28
- Cash Reserve: 15% (100) -> 30
- Readiness: (80) -> 24
- Penalty: Largest holding is 60% (-15)
- **Score**: 28 + 30 + 24 - 15 = **67**

### Why This Is Useful
- **Product**: Increases engagement by giving users a score to optimize.
- **Interview**: Shows ability to synthesize disparate data points into a cohesive, weighted business logic algorithm.

## SECTION 8 — PERFORMANCE TRACKING SYSTEM

### Architecture
```text
setInterval (Scheduler)
  ↓ (Triggers every 60s)
addSnapshotJob (snapshotQueue.js)
  ↓ (Pushes to Redis)
BullMQ Worker (snapshotWorker.js)
  ↓ (Consumes Job)
Fetch All Active Portfolios
  ↓ (Valuates using Finnhub cache)
MongoDB (PortfolioSnapshot.insertMany)
```

### Explanations
- **Why snapshots?**: Portfolios only store *current* state. We need snapshots to draw a historical chart.
- **Why queues (BullMQ)?**: If we have 10,000 users, taking snapshots takes time. If we did it on the main thread, the API would freeze. Queues offload this to a background process.
- **Why Redis?**: BullMQ uses Redis as an ultra-fast in-memory data structure store to manage job locks, retries, and state.

### Code Snippets (`snapshotWorker.js`)
```javascript
const worker = new Worker('portfolio-snapshots', async (job) => {
  const portfolios = await Portfolio.find({ $or: [{ 'holdings.0': { $exists: true } }, { cashBalance: { $ne: 10000 } }] });
  const snapshots = portfolios.map(p => {
    const valuation = valuatePortfolio(p, prices);
    return { userId: p.userId, portfolioValue: valuation.portfolioValue, ... };
  });
  await PortfolioSnapshot.insertMany(snapshots);
});
```
*Explanation*: The worker fetches all active portfolios, valuates them synchronously using the in-memory Finnhub price cache, constructs snapshot objects, and uses `insertMany` for a highly optimized bulk write to MongoDB.

## SECTION 9 — DATABASE DESIGN

### PortfolioSnapshot Collection
**Schema Fields:**
- `userId` (ObjectId, Ref: 'User'): Links the snapshot to the owner.
- `portfolioValue` (Number): Total value (cash + stocks).
- `cashBalance` (Number): Available cash.
- `holdingsValue` (Number): Value of all stocks.
- `timestamp` (Date): Exact time the snapshot was taken.

**Purpose**: 
To store historical, point-in-time valuations of a user's portfolio. Since stock prices fluctuate, we must lock in the valuation at regular intervals to draw a historical chart.

**Indexes**:
- `{ userId: 1, timestamp: 1 }`: A compound index. Queries always filter by `userId` and sort/filter by `timestamp`. This index makes the `getPerformance` query incredibly fast.

### Relationships
- **One-to-Many**: One `User` has Many `PortfolioSnapshots`.

## SECTION 13 — DBMS CONCEPTS USED

1. **Collections & Documents**: Using MongoDB (NoSQL) to store unstructured JSON-like data.
2. **References**: `userId` in `PortfolioSnapshot` is a reference to the `User` collection.
3. **Aggregation**: Used in `readinessController` to ` $group ` expenses by category and sum them up dynamically.
4. **Indexing**: A compound index `{ userId: 1, timestamp: 1 }` on Snapshots to optimize the time-series query for the performance chart.
5. **Bulk Operations**: `PortfolioSnapshot.insertMany()` used in the background worker to insert 10,000 snapshots in a single network request instead of 10,000 separate `save()` calls.

## SECTION 14 — SYSTEM DESIGN CONCEPTS

1. **AI Integration Layer**: Segregating LLM logic into a dedicated service. Treating AI as an unreliable 3rd party API that requires strict timeouts, schema enforcement, and robust fallbacks.
2. **Background Processing / Queue-Based Architecture**: Using BullMQ and Redis to handle heavy tasks (snapshot generation) asynchronously so the main Express event loop is never blocked.
3. **Event Driven Design**: The Finnhub WebSocket pushes events to `finnhubService`, which emits local Node `EventEmitter` events, which are consumed by the Socket.io server to push to clients.
4. **Service Layer Pattern**: Controllers only handle HTTP (req/res). Business logic (calculating readiness, calling Gemini, valuating portfolios) is strictly inside `/services`.
5. **Graceful Degradation**: If Gemini hits a rate limit (429), the system does not crash or show an error to the user. It gracefully degrades to a deterministic algorithmic fallback that mimics the AI perfectly.

### Backend Questions
1. **Q: Why use a Service layer instead of putting everything in the Controller?**
   - *A*: Separation of concerns. Controllers handle HTTP (status codes, req/res). Services handle business logic. This makes the code modular and unit-testable.
2. **Q: How does the snapshot worker handle 10,000 portfolios efficiently?**
   - *A*: By using `insertMany` for a single bulk network request to MongoDB, rather than looping and calling `.save()` 10,000 times.

### DBMS Questions
1. **Q: Why NoSQL (MongoDB) for portfolios instead of SQL?**
   - *A*: Portfolios have a dynamic array of holdings. MongoDB's document structure allows storing the entire portfolio in a single document without complex JOINs.
2. **Q: How do you optimize the performance chart query?**
   - *A*: By adding a compound index on `{ userId: 1, timestamp: 1 }`, ensuring the database can locate the user's snapshots and sort them in a single index scan.

### System Design Questions
1. **Q: Explain the queue-based architecture.**
   - *A*: A scheduler (setInterval) pushes a job to a Redis queue via BullMQ. A separate Worker process pulls the job and executes it. This decouples heavy lifting from the API server.
2. **Q: What is Graceful Degradation?**
   - *A*: Designing a system so that if a primary component fails (e.g., Gemini API), the application continues to function using a fallback (e.g., deterministic rules) rather than crashing.

*(Self-study note: Prepare 8 more SD questions focusing on Load Balancing, Horizontal Scaling, Caching, Rate Limiting, and Microservices vs Monoliths).*

2. **Q: What is the purpose of the dependency array in `useEffect`?**

### BullMQ / Redis Questions
1. **Q: Why do you need Redis for BullMQ?**

2. **Q: What happens if a snapshot job fails?**
